# FUT-K — Empirical Validation

This folder is the project's **scientific evidence**: what was tested, on which
data, with which methodology, what the numbers are, and how to reproduce every
one of them. Raw generated reports live in [`results/`](./results/); this
document is the guided tour.

FUT-K's own philosophy (design doc, Section 20) applies here first:
**calibration is the judge**, and a negative result honestly reported is worth
more than a positive one that cannot be trusted. Two experiments below are
negative results, kept on purpose.

---

## 1. Maturity level

Using the standard four-level ladder for systems like this:

| Level | Meaning | Status |
|---|---|---|
| 1. Conceptual architecture | the model is defined | ✅ complete (`docs/design/`) |
| 2. Implementation | the model is software | ✅ complete (engine + app) |
| 3. Internal validation | algorithms provably do what the spec says | ✅ complete (§2 below) |
| 4. External validation | performance measured on real competitions | 🟡 **in progress** — real-data calibration on 3 competitions done (§3–§5); larger samples and external baselines (Elo, market odds) still open (§7) |

We deliberately do **not** claim level 4 is finished. What exists at level 4
today is: leakage-safe backtests on 462 real matches across three competitions,
walk-forward parameter fitting with honest held-out metrics, and a baseline
comparison against the constant base-rate predictor. What does not exist yet is
listed in §7.

## 2. Internal validation (synthetic, ground-truth known)

Every mechanism in the engine is tested against synthetic data with a *known*
generator, so pass/fail is objective. The full plan is
[`docs/design/validation_test_plan.md`](../docs/design/validation_test_plan.md)
— 89 numbered test IDs (`T-SS-NN`), all implemented, plus property-based tests
(hypothesis) and multi-seed Monte-Carlo runs. **189 engine tests + 18 API tests
green in CI** (Python 3.11 & 3.12).

Reference numbers reproduced (see [`results/RESULTS.md`](./results/RESULTS.md),
regenerated by `scripts/report.py`, never hand-edited):

| Quantity | Target | Observed | Test ID |
|---|---|---|---|
| Goals/match at `BASE_RATE=0.015` (80k sims) | ~2.7 | 2.688 | T-08-06 |
| Brier of a perfectly calibrated predictor | ~0.190 | 0.192 | T-20-01 |
| Reliability deviation of calibrated predictor | ~0.001 | 0.003 | T-20-03 |
| Source-credibility convergence | 0.70 | 0.6999 | T-15-04 |
| Narrative-memory convergence | 0.18 | 0.1778 | T-15-09 |
| **No information leakage in `backtest()`** | byte-identical | ✅ hard gate | T-20-04 |

The leakage test (T-20-04) is the foundation of everything in §3–§5: a
prediction made at minute *t* is proven to be byte-for-byte identical whether
or not future events exist in the record — and the same property is re-tested
at the HTTP level of the app's API.

## 4. Backtest methodology

- **Replay, never foresight.** Each match is replayed minute by minute; the
  prediction at minute *t* uses only events with `minute ≤ t` (enforced by the
  T-20-04 gate). Target variable: *"at least one goal in the next 10 minutes"*,
  evaluated every 5 minutes.
- **Walk-forward fitting.** Parameters (`base_rate`, `tau`) are always fit on
  matches strictly **before** (by date) the matches they are scored on —
  expanding training window, contiguous future test block, 4 folds. Reported
  metrics are held-out only.
- **Metrics.** Brier score, log loss, reliability curve (10 bands), and the
  calibration gap `|mean predicted − observed frequency|`.
- **Seeds.** Every stochastic test runs on ≥3 fixed seeds; CI uses a pinned
  seed for reproducibility.

## 5. Results on real data

### 5.1 Calibration (World Cup 2018)

From [`results/RESULTS_FIT.md`](./results/RESULTS_FIT.md) — 64 matches, 4-fold
walk-forward, held-out only. Baseline = the constant predictor that always
outputs the observed base frequency (0.236):

| Predictor | Mean predicted | Brier ↓ | Log loss ↓ |
|---|---|---|---|
| Constant base-rate baseline | 0.236 | 0.1803 | 0.5464 |
| Engine, untuned | 0.275 | 0.1797 | 0.5452 |
| **Engine, walk-forward fitted** | **0.211** | **0.1796** | **0.5449** |

Honest reading: the untuned engine **over-predicted** (0.275 vs 0.236 observed);
fitting closed the calibration gap from 0.040 to 0.025 out of sample, and the
fitted engine beats the constant baseline — **narrowly**. On this target and
sample size, in-play state adds a small, real, but modest amount of signal
beyond the base rate. That is the truthful current state, not a limitation we
hide.

### 5.2 The model distinguishes competitions

From [`results/RESULTS_COMPARE.md`](./results/RESULTS_COMPARE.md) — fit each
competition, cross-apply to the others:

| Competition | Goals/90 | Fitted base_rate | Fitted tau |
|---|---|---|---|
| Champions League finals | 2.70 | 0.013 | 4 |
| World Cup 2018 | 2.46 | 0.012 | 4 |
| La Liga 2015/16 (50-match sample) | 2.28 | 0.012 | 16 |

Finals score more per 90 minutes than regular league games, and each
competition's own fitted parameters beat the others' on its own data (full
cross table in the report). League play also prefers a much longer pressure
memory (`tau` 16 vs 4) than knockout football.

### 5.3 Negative result: per-regime calibration does not help

From [`results/RESULTS_REGIME.md`](./results/RESULTS_REGIME.md): adding a
per-regime λ multiplier **worsens** held-out log loss (0.5449 → 0.5498), and on
the full data every regime's best scale is ~1.0 — the base multipliers already
encode what the regime label would add. Kept as designed-in honesty (design
doc, Section 24).

### 5.4 Negative result: a learned model does not beat the heuristic

From [`results/RESULTS_LEARNED.md`](./results/RESULTS_LEARNED.md): a logistic
regression over leakage-safe state features loses to the fitted heuristic out
of sample (log loss 0.5550 vs 0.5449). Per the design doc (Section 21), a
learned model is only adopted if it wins out of sample — it did not, so it was
not adopted.

### 5.5 Player-level layers (face validity on real data)

- **Profiles** ([`results/RESULTS_PROFILES.md`](./results/RESULTS_PROFILES.md)):
  603 World Cup players; top scorers = Kane, Cheryshev, Ronaldo, Lukaku (all
  labeled `finisher`), top key-pass creators = Neymar, Özil, Müller (all
  `creator`) — the archetypes match football reality.
- **Passing network** ([`results/RESULTS_NETWORK.md`](./results/RESULTS_NETWORK.md)):
  Barcelona 2015/16 recovers its real structure — robustness 0.933, most
  central node Busquets, strongest link Alba→Neymar (575 passes).
- **On/off influence** ([`results/RESULTS_INFLUENCE.md`](./results/RESULTS_INFLUENCE.md)):
  kept as a **worked example of confounding** — naive on/off inverts (starters
  rested during blowouts look "negative") — exactly the trap the design doc
  warns about; presented as such, not as a player ranking.

## 6. Reproducibility

Everything below runs offline-first (synthetic suites) or against the free
StatsBomb open data (downloaded on demand into `.sb_cache/`, ~1.1 GB for the
full set).

```bash
# 0. environment
pip install -e ".[dev]"                      # engine + test deps

# 1. internal validation (no network needed)
pytest -q                                     # 189 tests, incl. slow MC
python scripts/report.py                      # regenerates results/RESULTS.md

# 2. real-data calibration + fitting (downloads WC 2018 on first run)
python scripts/ingest_statsbomb.py --competition 43 --season 3 --limit 64
python scripts/fit_statsbomb.py    --competition 43 --season 3 --limit 64 --folds 4

# 3. competition comparison (downloads CL finals + La Liga sample)
python scripts/compare_competitions.py

# 4. per-regime calibration (negative result)
python scripts/fit_regime_statsbomb.py --limit 64 --folds 4

# 5. learned model vs heuristic (negative result)
python scripts/fit_learned_statsbomb.py --limit 64 --folds 4

# 6. player layers on real data
python scripts/build_profiles.py  --competition 43 --season 3 --limit 64
python scripts/build_networks.py  --team Barcelona
python scripts/build_influence.py --team Barcelona
```

Each script rewrites its report in `validation/results/` — the committed
reports are exactly what these commands produce. Seeds are fixed in the code;
the StatsBomb dataset is versioned upstream by its git history.

## 7. What is NOT validated yet (open work)

Transparency about the gap between here and full external validation:

- **No comparison against external baselines** — Elo ratings, bookmaker odds,
  or public xG models. This is the single most important missing benchmark.
- **No betting-market metrics** (ROI, accuracy vs closing lines). Note the
  project's stated goal is *understanding*, not betting — but the comparison
  would still anchor the calibration externally.
- **Single-target evaluation.** Only *goal-in-next-10-minutes* has been scored
  at scale; corners/cards/next-scorer targets exist in the engine but have no
  real-data calibration report yet.
- **Sample size.** 462 matches is enough to measure calibration honestly, not
  enough to resolve small effects (the competition differences in §5.2 are
  consistent but small).
- **No live-stream validation** — all real-data work is historical replay. The
  latency and consensus layers of the design doc (Sections 4, 16) are not yet
  exercised against live feeds.
